# Open3DStream Serialization Frame Size Analysis

## MOCAP (SubjectList) SERIALIZATION

### Structure Overview
- Uses Google FlatBuffers for efficient binary serialization
- Wire format: 4-byte flags + 4-byte CRC32 = 8 bytes overhead
- Per Subject: name string (10-50 bytes) + format string (5-20 bytes) + axis directions (3 bytes) + overhead (10-20 bytes) = ~30-100 bytes
- Per Transform Node: name (5-20 bytes) + parent ID (4 bytes) + component flags (1-4 bytes) + overhead (~8 bytes) = ~20-40 bytes overhead
- Per Transform Data: Translation (3 floats = 12 bytes) + Rotation (4 floats = 16 bytes) + Scale (3 floats = 12 bytes) = 40 bytes

### Typical Mocap Frame Sizes

Standard humanoid skeleton (50 bones):
- Fixed overhead: 8 bytes
- Subject metadata: 50 bytes
- 50 Transform node headers: 50 x 30 = 1,500 bytes
- 50 Transform TRS data: 50 x 40 = 2,000 bytes
- FlatBuffer headers/padding: ~400 bytes
- TOTAL: ~4,000 bytes (4 KB)

Range by complexity:
- Minimal (10 bones): 800-1,200 bytes
- Small (30 bones): 2,000-3,000 bytes
- Standard (60 bones): 4,000-6,000 bytes
- Complex with curves: +500-2,000 bytes

### Mocap Data Rate
30 FPS x 4 KB = 122.88 kbps per stream


## AUDIO SERIALIZATION

### Unified Message Header (all audio types)
- Magic: 4 bytes
- Version: 1 byte
- Kind: 1 byte
- Codec: 1 byte
- Flags: 1 byte
- Timestamp: 8 bytes
- Payload size: 4 bytes
- TOTAL: 20 bytes

### PCM16 Audio Metadata (uncompressed)
- Version: 1 byte
- Flags: 1 byte
- Channels: 2 bytes
- Sample Rate: 4 bytes
- Timestamp: 8 bytes
- SourceGuid: 16 bytes
- LabelSize: 2 bytes
- SubjectSize: 2 bytes
- PCMBytes: 4 bytes
- TOTAL HEADER: 40 bytes
- Typical strings (StreamLabel + SubjectName): ~18 bytes

### PCM16 Frame Calculation
48 kHz stereo, 20ms duration:
- Samples: 960 (48000 x 0.020)
- PCM16 data: 960 x 2 channels x 2 bytes = 3,840 bytes
- Metadata overhead: 40 bytes (fixed) + 18 bytes (strings) = 58 bytes
- Unified header: 20 bytes
- TOTAL FRAME: 3,918 bytes (~3.8 KB)
- 50 FPS audio (20ms frames): 3,918 x 50 = 195.9 kbps

### Opus Encoded Audio
- Same metadata structure as PCM16 but with codec field (42 byte header)
- 64 kbps bitrate encoding: 160 bytes (64000 bits/sec x 0.020 sec / 8)
- Typical metadata: 60 bytes (42 header + 18 strings)
- Total: 160 + 60 + 20 unified header = 240 bytes per frame
- 50 FPS audio: 240 x 50 = 12 kbps (actual codec) + 20 kbps overhead = 84 kbps total

### Compression Ratios (Opus vs PCM16)
- 64 kbps Opus: 160 / 3,840 = 4.2% of PCM16 size
- 128 kbps Opus: 320 / 3,840 = 8.3%
- 32 kbps Opus: 80 / 3,840 = 2.1%

### Overhead Analysis
- PCM16: 78 bytes overhead in 3,918 byte frame = 2% overhead
- Opus 64kbps: 80 bytes overhead in 240 byte frame = 33% overhead (proportionally higher due to smaller payload)


## COMBINED MOCAP + AUDIO SCENARIOS

### Scenario 1: Mocap Only
- 30 FPS, 4 KB frames = 122.88 kbps
- Data/sec: 15.36 KB

### Scenario 2: Audio PCM16 Only
- 50 FPS, 3.9 KB frames = 195.9 kbps
- Data/sec: 24.49 KB

### Scenario 3: Audio Opus 64kbps Only
- 50 FPS, 240 byte frames = ~84 kbps
- Data/sec: 10.5 KB

### Scenario 4: Mocap + Audio PCM16
- Mocap: 123 kbps
- Audio PCM16: 196 kbps
- TOTAL: 319 kbps (39.85 KB/sec)

### Scenario 5: Mocap + Audio Opus 64kbps (OPTIMIZED)
- Mocap: 123 kbps
- Audio Opus: 84 kbps
- TOTAL: 207 kbps (25.86 KB/sec)
- SAVINGS vs PCM16: 35% bandwidth reduction


## KEY SIZE DETERMINANTS

Mocap Drivers (SubjectList):
1. Number of bones (primary) - linear, ~40-70 bytes per bone
2. String lengths - bone/subject names
3. Skeleton complexity - curves, custom matrices
4. FlatBuffer overhead - fixed ~400-600 bytes

Audio Drivers:
1. Codec selection (primary) - PCM16 = 100%, Opus = 4-10% of uncompressed
2. Sample rate - 48kHz vs 44.1kHz vs 16kHz
3. Channels - mono, stereo, surround
4. Frame duration - longer frames = more data per transmission
5. String metadata - negligible vs audio data
