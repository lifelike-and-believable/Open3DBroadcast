================================================================================
                 OPEN3DSTREAM FRAME SERIALIZATION ANALYSIS
================================================================================

MOCAP (SubjectList) FRAMES
================================================================================

STRUCTURE:
  CRC32 Header:          8 bytes (flags + CRC)
  Subject per frame:     50-100 bytes (name, format, directions)
  Per bone overhead:     20-40 bytes (name, parent ID, components, FlatBuffer)
  Per bone data:         40 bytes (Translation 12B + Rotation 16B + Scale 12B)

TYPICAL SIZES:
  10 bones:              800-1,200 bytes
  30 bones:              2,000-3,000 bytes
  50 bones (STANDARD):   4,000 bytes (~4 KB) <-- Industry typical
  60 bones:              4,000-6,000 bytes

BANDWIDTH (at 30 FPS):
  Standard skeleton:     4 KB x 30 = 122.88 kbps (15.36 KB/sec)

WHAT DRIVES SIZE:
  PRIMARY:   Number of bones (linear ~40-70 bytes/bone)
  SECONDARY: String lengths, curves, custom matrices
  TERTIARY:  FlatBuffer overhead (~400-600 bytes fixed)


AUDIO FRAMES
================================================================================

PCM16 (UNCOMPRESSED)
─────────────────────────────────────────────────────────────────────────────
  Unified Header:        20 bytes (magic, version, codec, timestamp, size)
  Audio Metadata:        40 bytes (channels, sample rate, guid, sizes)
  Strings (typical):     18 bytes (StreamLabel + SubjectName)
  Audio data (48kHz, 2ch, 20ms): 960 samples x 2 channels x 2 bytes = 3,840 bytes
  
  TOTAL FRAME:           3,918 bytes (~3.8 KB)
  OVERHEAD:              78 bytes (2% of total)

  Bandwidth at 50 FPS:   3,918 x 50 = 195.9 kbps (24.49 KB/sec)


OPUS ENCODED (64 kbps bitrate)
─────────────────────────────────────────────────────────────────────────────
  Unified Header:        20 bytes
  Audio Metadata:        42 bytes (includes codec field)
  Strings (typical):     18 bytes
  Encoded audio:         160 bytes (64000 bits/sec x 0.020 sec / 8)
  
  TOTAL FRAME:           240 bytes (~0.24 KB)
  OVERHEAD:              80 bytes (33% of total) *proportionally higher*

  Bandwidth at 50 FPS:   240 x 50 = 12 kbps (codec data) + 20 kbps (overhead) = 84 kbps total
  COMPRESSION vs PCM16:  160 / 3,840 = 4.2%

OPUS BITRATE COMPARISON:
  32 kbps:   80 bytes encoded -> 160 byte total frame -> 2.1% vs PCM16
  64 kbps:   160 bytes -> 240 byte total frame -> 4.2% vs PCM16 [STANDARD]
  128 kbps:  320 bytes -> 380 byte total frame -> 8.3% vs PCM16


COMBINED MOCAP + AUDIO SCENARIOS
================================================================================

Scenario 1: MOCAP ONLY
  Bandwidth: 122.88 kbps
  Data/sec:  15.36 KB

Scenario 2: AUDIO ONLY - PCM16 (No compression)
  Bandwidth: 195.9 kbps
  Data/sec:  24.49 KB

Scenario 3: AUDIO ONLY - Opus 64 kbps
  Bandwidth: 84 kbps
  Data/sec:  10.5 KB
  Savings vs PCM16: 57%

Scenario 4: MOCAP + AUDIO PCM16 (UNOPTIMIZED)
  Mocap:     122.88 kbps
  Audio:     195.9 kbps
  TOTAL:     318.78 kbps
  Data/sec:  39.85 KB

Scenario 5: MOCAP + AUDIO Opus 64 kbps (OPTIMIZED)
  Mocap:     122.88 kbps
  Audio:     84 kbps
  TOTAL:     206.88 kbps
  Data/sec:  25.86 KB
  SAVINGS vs Scenario 4: 35%


OVERHEAD BREAKDOWN
================================================================================

MOCAP Frame (4 KB total):
  Actual skeleton data:  ~2,400 bytes (TRS for 50 bones)
  Node overhead:        ~1,500 bytes (names, parent refs, FlatBuffer)
  Subject metadata:     ~50 bytes
  Wire header:          ~8 bytes
  Other (padding):      ~400 bytes
  ───────────────────────────────
  Overhead percentage: ~45% of total frame


AUDIO PCM16 Frame (3.9 KB total):
  Audio PCM data:       3,840 bytes (98% of payload)
  Metadata headers:     60 bytes
  Unified header:       20 bytes
  ───────────────────────────────
  Overhead percentage: 2% of total frame


AUDIO OPUS Frame (240 bytes total):
  Encoded audio:        160 bytes (67%)
  Metadata headers:     62 bytes
  Unified header:       20 bytes
  ───────────────────────────────
  Overhead percentage: 33% of total frame (high for small payloads)


KEY FINDINGS
================================================================================

1. MOCAP IS SIZE-EFFICIENT:
   - ~4 KB per 50-bone frame is minimal for skeletal data
   - Overhead is ~45% due to FlatBuffer structure and per-bone metadata
   - Scales linearly: each bone adds ~40-70 bytes

2. AUDIO COMPRESSION IS CRITICAL:
   - PCM16 at 48kHz/stereo = 196 kbps (huge)
   - Opus at 64 kbps = 84 kbps (4.2% of PCM16 encoded size)
   - Using Opus saves 35% bandwidth in combined mocap+audio scenario

3. OVERHEAD PATTERNS:
   - PCM16: Very low overhead (2%) due to large payload
   - Opus: High proportional overhead (33%) due to small payload
   - Mocap: Moderate overhead (45%) due to FlatBuffer structure

4. BANDWIDTH ESTIMATES:
   Mocap-only:           ~123 kbps
   Audio PCM16:          ~196 kbps
   Audio Opus 64k:       ~84 kbps
   Combined PCM16:       ~319 kbps
   Combined Opus:        ~207 kbps (best case)


OPTIMIZATION RECOMMENDATIONS
================================================================================

For LAN (10+ Mbps):
  Use PCM16 audio, no optimization needed
  Standard settings work fine

For Local Network (1-10 Mbps):
  Use Opus @ 64 kbps for audio
  Keep mocap at 30 FPS
  Results: ~207 kbps total, well within limits

For Internet (< 1 Mbps upstream):
  Use Opus @ 32 kbps for audio
  Reduce mocap to 15 FPS if needed
  Results: ~90 kbps mocap + 42 kbps audio = ~132 kbps

For Constrained Bandwidth:
  Mocap-only mode (123 kbps)
  or
  Opus @ 16 kbps + 15 FPS mocap (75 kbps combined)

SINGLE MOST IMPACTFUL CHANGE:
  Switch from PCM16 to Opus encoding for audio (35% bandwidth savings)
